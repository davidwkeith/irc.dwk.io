# Design rationale — `irc://`

Generated by Anglesite design-interview, hand-tuned for the Gen-X-to-Gen-Z register described in the project CLAUDE.md and PRD. This document explains every visible decision and how to change any of it.

## Posture

The site is the product/infrastructural face of `irc://` — not the campaign URL (which lives separately, with a different and more activist voice per CLAUDE.md). Reference set: [htmx.org](https://htmx.org), [tantek.com](https://tantek.com), [aaronparecki.com](https://aaronparecki.com), [pluralistic.net](https://pluralistic.net), [modern.ircdocs.horse](https://modern.ircdocs.horse), the IETF datatracker, [htmhell.dev](https://htmhell.dev). All hit the same chord — visible structure, mono everywhere, view-source as a feature, no shadows or gradients, links that look like links, paragraphs that read like blog posts not landing-page copy.

The posture is "the elders know how this works, the kids see through corporate polish, and the meeting point is craft-over-flash." Authority through craft, not through marketing-page formality.

## Axes

| Axis | Position | Reading | Why |
| --- | --- | --- | --- |
| Temperature | 0.40 | leaning cool | Civic/RFC heritage pulls cool. Warmth comes from typography and tone, not color. |
| Weight | 0.50 | balanced | Spec text needs breathing room; structure needs density. Middle ground. |
| Register | 0.60 | leaning authoritative | Trust-as-infrastructure is the brand pitch, but craft beats polish. |
| Time | 0.20 | strongly classic | RFC + textfiles + IndieWeb heritage all pull this way. Not retro — leaning hard but not anchored. |
| Voice | 0.40 | leaning subtle | The wordmark `irc://` carries volume; let it. Single-color accent. |

## Palette

Single-accent amber is the only color; everything else is near-black on warm cream paper for light mode, warm off-white on warm near-black for dark mode. The amber gestures at CRT phosphor and IRC's 1988 origin without costuming.

| Token | Light | Dark | Used for |
| --- | --- | --- | --- |
| `--color-brand` | `#8c4f00` | `#ffb000` | Body links, interactive elements. Light value chosen to pass WCAG AA on the cream bg. Dark value is full phosphor amber. |
| `--color-accent` | `#c77a1a` | `#ffc940` | Decorative spots — large headers, highlight rules. Not for body text in light mode (contrast-thin for that role). |
| `--color-bg` | `#f8f4ed` | `#0f0e0c` | Page background. Warm cream / warm near-black. |
| `--color-surface` | `#f1ece0` | `#1a1714` | Card/code-block surface — slightly more pigment than bg. |
| `--color-text` | `#1a1614` | `#f0e8d8` | Body text, headers. Warm undertone keeps it from feeling printer-cold. |
| `--color-muted` | `#5c544e` | `#9a9085` | Secondary text — captions, timestamps, meta. |
| `--color-border` | `#d4ccbc` | `#3a342c` | Hairline rules. Used everywhere a divider is needed. |

WCAG AA contrast verified: `--color-text` on `--color-bg` is ~14:1 (well above the 4.5:1 normal-text requirement); `--color-muted` on `--color-bg` is ~7:1 (above the 4.5:1 threshold for body text); `--color-brand` on `--color-bg` is ~4.7:1 (passes 4.5:1 normal-text). Dark mode is verified analogously.

Want to change the accent? Replace `#8c4f00` and `#ffb000` in `src/design/tokens.css`. The single-color discipline is part of the aesthetic — adding a second accent (e.g., a complementary cool color) is a real change that should be deliberated, not snuck in.

## Typography

**Mono everywhere.** Display, headers, body, code, nav — all one stack:

```css
ui-monospace, "SF Mono", Menlo, Consolas, "Cascadia Code", "Liberation Mono", monospace
```

System fonts only per [ADR-0005](../../docs/decisions/0005-system-fonts.md). On Apple platforms `ui-monospace` resolves to SF Mono. On Windows we fall through to Cascadia Code or Consolas. On Linux we land on Liberation Mono or whatever the user's system mono is. No CDN, no FOUT, no GDPR-relevant external font loading.

**Hierarchy via size and spacing, not via font change.** Headers are larger and more spaced; body is base-size with looser line-height. The typography type scale uses 1.25 ratio (modest minor-third). Reading measure is `64ch` — narrower than serif/sans norms because monospace fits fewer characters per em, and the narrow measure preserves line-length comfort.

**Line heights** are tuned for mono — slightly denser than serif/sans (`--line-height-base: 1.55`) because monospace already has more horizontal space per character.

If reading-length tests on the spec pages turn up problems, the cheapest pivot is to swap `--font-body` to `ui-serif, "New York", Georgia, "Iowan Old Style", serif` in tokens.css and accept the looser register. Don't introduce a sans-serif anywhere — it would push the whole register toward contemporary/corporate and conflict with every other axis position.

## Shape

**Zero radius, no shadows.** Every `--radius-*` is `0`; every `--shadow-*` is `none`. Cards are bounded by hairline borders (`--border-hairline`); code blocks have a hairline rule above and below; the directory's filter UI has hairline horizontal rules. This is the brutalist-but-readable register — htmx and IndieWebCamp both treat shape this way.

If a future component genuinely needs a radius (the only place I can see is OS-native form controls that look weird without a slight radius), introduce `--radius-control: 2px` rather than reaching for the existing tokens. Don't broaden the discipline without a reason.

## Spacing

The spacing scale is the design.ts default: `xs 0.25rem / sm 0.5rem / md 1rem / lg 2rem / xl 4rem`. With `weight = 0.5` the multiplier is 1.0, so values are unscaled. Layouts use `md` for component padding, `lg` between major sections, `xl` between page-level blocks. The `xs` and `sm` are for tight rhythm — gap between a label and its value, padding inside a small chip.

## What this design rules out

To make the discipline real and easy to recognize when it's slipping:

- No sans-serif anywhere
- No gradient
- No box-shadow
- No `border-radius` beyond hairline
- No "Join the community" CTA-button styling
- No animations beyond `:focus` rings and link hover (color change only)
- No background-color hero block
- No stock-photography hero
- No background image of any kind on body or main containers
- No icon library beyond what we draw inline as monospace ASCII or simple SVG
- No second accent color
- No Tailwind/Material/Bootstrap defaults

## What this design embraces

- ASCII separators (`---`, `===`) as visual elements
- `Last-Modified:` timestamps in RFC-style on every page
- Microformats visible in view-source (`h-card`, `e-content`)
- Code samples that look like code samples
- Single-column reading
- Hairline borders everywhere a divider is needed
- View-source-friendly markup — semantic HTML, no class-soup, no div-divitis
- Skip links and focus rings as first-class, not afterthoughts
- Spec docs that read like RFCs because they essentially are RFCs

## How to override

All design decisions are CSS custom properties in `src/design/tokens.css`. Change the file, save, and the change is global. No build step required for token changes. Components in `src/components/` consume the tokens; if you find a hardcoded color or font in a component, that's a bug — file it and fix it.

To regenerate the design wholesale via the design-interview skill, run `/anglesite:design-interview` and pick "start fresh" when asked. To nudge axes incrementally, run `/anglesite:design-interview` and pick "update specific parts."
